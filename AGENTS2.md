# AGENTS.md — MediStock

> This file is a **scope guard** for AI/agents working in this repository.
> Applies to Codex CLI, Claude Code, or any other agent reading the repo.

## Project Identity

- **Name**: MediStock
- **Type**: Joki / MVP demo
- **Target**: Android pharmacy inventory app
- **Stack**:
  - Mobile: Flutter (Android-first), GetX, feature-based
  - Backend: NestJS (modules-based, vertical slice)
  - Database: PostgreSQL
- **Sub-projects**:
  - `medistock_mobile/` — Flutter app
  - `medistock-api/` — NestJS backend
  - `docs/` — single source of truth for product/technical decisions
  - `ui/` — mockups and design assets

## Scope Rules (MANDATORY)

### 1. Single Source of Truth

All product/technical decisions **must** reference documents in `docs/`. Authority order on conflict:

1. `docs/prd.md` — what to build (features, flows, constraints)
2. `docs/api_contract.md` — endpoint contracts, request/response, status codes
3. `docs/database_schema_mvp.md` — DB structure used by the MVP
4. `docs/folder_structure.md` — file locations and architecture style

`docs/database_schema_full.md` is **NOT** in scope for the MVP phase. It exists for future reference only and must not be referenced as authority during MVP work.

### 2. Hard Rules (Do Not Violate)

- **Do not** add dependencies, libraries, or services beyond what is listed in `docs/`.
- **Do not** write code for third-party platforms (iOS production, web, desktop) unless explicitly requested. iOS/macOS/windows/linux folders generated by `flutter create` **should be left as-is** (default scaffolding, not implementation targets).
- **Do not** deviate from the architecture style defined in `docs/folder_structure.md`:
  - Backend: modules-based NestJS, one module per domain feature.
  - Mobile: feature-based with GetX.
- **Do not** create endpoints that are not in `docs/api_contract.md`.
- **Do not** add tables/columns outside `docs/database_schema_mvp.md`.
- **Do not** perform work outside the **Out-of-Scope** list below.

### 3. Out-of-Scope (Agent May Refuse or Ask First)

The agent **may refuse or ask for confirmation** before doing any of the following:

- Premature optimization (complex caching, fine-grained indexing) without measured need.
- Major refactors / deeply layered clean architecture.
- CI/CD setup, production-grade Docker, deployment automation.
- Heavy e2e or integration testing. Unit tests on controllers/services are allowed.
- Multi-language internationalization (i18n).
- Push notifications, payment gateways, external integrations.
- Migration to `database_schema_full.md` (batch, PO, sales, opname, audit, etc.).
- Complex authentication (OAuth, SSO, 2FA). Keep it simple per PRD.

### 4. Pre-Work Checklist

Before starting any task, the agent **must**:

1. Read `docs/prd.md` for feature context.
2. Read `docs/folder_structure.md` to know where to place files.
3. If touching endpoints → read `docs/api_contract.md`.
4. If touching DB/ORM/entities → read `docs/database_schema_mvp.md`.
5. If touching UI → open the mockups in `ui/`.

If a user request **conflicts** with `docs/*.md`, the agent must **ask first** before changing code, and recommend updating the document first.

## Code Conventions (Brief)

- **Backend (NestJS)**:
  - Language: TypeScript.
  - ORM/DB: per `docs/database_schema_mvp.md` (Prisma/TypeORM to be decided at implementation — pick the fastest path for MVP).
  - Validation: `class-validator` with DTOs on every endpoint.
  - Error response: consistent with the format in `docs/api_contract.md`.
- **Mobile (Flutter)**:
  - Language: Dart.
  - State management: GetX.
  - HTTP client: use whatever is already in `pubspec.yaml` (check before adding).
  - Feature folder: `lib/features/<feature_name>/` containing `controllers/`, `views/`, `bindings/`, `models/`.
- **Format & Lint**:
  - Backend: `prettier` + `eslint` (config already in `medistock-api/`).
  - Mobile: `flutter analyze` + built-in formatter.
  - Do not impose a style guide beyond the default tool configs above.
